## 1. Purpose

This file provides a **redacted sample** of `admin_jit_escalations.business_justification` entries from FORM's PAM system, prepared for SOC 2 auditor review.

Per DEC-043, `business_justification` (plain-text) is excluded from bulk evidence exports because it may carry customer-identifying context (tenant names, support ticket IDs, Linear issue URLs). Instead, auditors receive:

1. **This redacted sample** — representative justifications sanitised to remove all PII and tenant-identifying details, demonstrating that the control operates as intended (meaningful justifications are required, recorded, and preserved).
2. **`justification_hash`** (SHA-256) in the DEC-030 HMAC chain — the cryptographic binding between the live production record and the immutable audit log.
3. **Hash verification on request** — for any sample entry, the auditor may request hash verification against the live `admin_jit_escalations` row under a time-limited `security_reviewer` grant on a staging replica. No bulk plaintext disclosure is required.

## 2. Redaction Methodology

| Field | Exported to auditor | Redaction rule |
|---|---|---|
| `pam_session_id` | ✅ (anonymised placeholder) | Original UUIDs replaced with sequential tokens (`PAM-SAMPLE-NNN`) |
| `access_level` | ✅ (verbatim) | No redaction |
| `escalation_start` | ✅ (date-only, no time) | Time-of-day removed; year/month retained for pattern verification |
| `escalation_expiry` | ✅ (TTL class only: `4h` / `1h` / `15m`) | Absolute timestamp removed |
| `status` | ✅ (verbatim) | No redaction |
| `is_break_glass` | ✅ (verbatim) | No redaction |
| `target_tenant_id` | ❌ Redacted → `[TENANT-REDACTED]` | Real UUID removed; tenant identity withheld |
| `actor_user_id` | ❌ Redacted → `[ADMIN-REDACTED]` | Real UUID removed; FORM staff identity withheld from bulk export |
| `business_justification` | ✅ (sanitised) | Customer names, tenant names, ticket IDs, Linear URLs replaced with `[REDACTED]` |
| `justification_hash` | ✅ (verbatim SHA-256) | No redaction — this is the cross-reference to DEC-030 chain |
| `authorised_by_ic_user_id` | ❌ Redacted → `[IC-REDACTED]` | Approver identity withheld |

### 3.4 Break-Glass Access (`is_break_glass = true`)

Break-glass sessions bypass the standard JIT approval workflow — they are initiated directly via the `form-break-glass` Cloudflare Access app. `authorised_by_ic_user_id` is NULL at session start (populated post-hoc when the 72-hour review is completed). Every break-glass activation emits a `pam.break_glass_activated` CRITICAL DEC-030 event and triggers an immediate PagerDuty P0 alert (AL-PAM-01).

A `pam_break_glass_reviews` row must be created and `pam.break_glass_review_completed` emitted within 72 hours (enforced by AL-PAM-BG-01).

## 4. Hash Verification Protocol

Auditors may request plaintext verification for any sample entry during fieldwork:

1. Provide the `justification_hash` (SHA-256) from this document.
2. FORM's compliance-officer will compute `SHA-256(business_justification)` from the live `admin_jit_escalations` row on a point-in-time staging replica with `security_reviewer` read grant.
3. The computed hash is compared against the value in this document and the value in the corresponding DEC-030 `pam.elevation_requested` event (`justification_hash` field).
4. All three values must match — this proves the justification was recorded before access was granted (causal order enforced by `pam-elevation-service` write sequencing in `docs/DATA_MODEL.md §29.2`).
5. The plaintext justification is shared verbally or under short-lived secure link — never in a bulk export file.

## 6. Limitations and Auditor Notes

- **Sample is not exhaustive.** This set was manually selected to represent the range of access levels, tenant scopes, and justification quality. The full population of `admin_jit_escalations` rows is available via SQL query under a time-limited `security_reviewer` grant.
- **Hashes in this document are representative only.** The SHA-256 values shown are structure-preserving placeholders for this redacted document. Verification against live production hashes requires the hash-verification protocol in §4.
- **No session IDs map to production.** The `PAM-SAMPLE-NNN` and `PAM-SAMPLE-BG-NNN` tokens are not `pam_session_id` UUIDs. Specific session correlation for fieldwork verification is handled via the hash-verification protocol.
- **Art. 9 data never appears in justifications.** FORM's style guide for justification text (`docs/SSO_SCIM_IMPLEMENTATION.md §24.3`) prohibits including individual health data, biometric values, or personal performance data in the `business_justification` field. Justifications describe the operational need, not the data content.
