Infrastructure Security

FORM runs on Cloudflare Workers (edge compute), Supabase (managed Postgres), and Cloudflare R2 (object storage). The underlying infrastructure is operated by these sub-processors, each holding SOC 2 Type II and ISO 27001 certifications.

§3.1 · Physical infrastructure
Control FORM Shared Customer
Data centre physical security Cloudflare / Supabase data centres — SOC 2 certified. FORM does not operate physical infrastructure.
Hardware lifecycle and disposal Sub-processor responsibility (Cloudflare, Supabase).
Geographic region selection FORM assigns tenants to EU (Frankfurt) or US (Virginia) based on contract. Customer specifies region requirement during onboarding.
§3.2 · Network security
Control FORM Shared Customer
DDoS mitigation Cloudflare enterprise DDoS protection on all FORM endpoints.
TLS termination TLS 1.2 minimum, TLS 1.3 preferred. HSTS with max-age=31536000; includeSubDomains; preload.
API rate limiting Per-tenant and per-user rate limits enforced at Worker layer. 500-request overage grace before hard block (DEC-045).
Customer network (egress from IdP to FORM) Customer ensures outbound HTTPS is permitted from their IdP to FORM SSO endpoints. Required IP ranges documented in docs/SSO_CLIENT_CONFIG.md.
IP allowlisting for admin dashboard FORM provides the technical control (session_policy.ip_allowlist). Customer configures the allowlist during onboarding.
§3.3 · Platform patching
Control FORM Shared Customer
Cloudflare Workers runtime updates Cloudflare manages runtime. FORM monitors for deprecations.
Supabase Postgres version upgrades FORM owns the upgrade schedule and tests migrations in staging before production.
Application dependency updates Dependabot + weekly security patch sprint. Critical CVEs patched within 72 hours per docs/VULNERABILITY_MANAGEMENT.md.
Mobile app updates (iOS / Android) FORM publishes updates to App Store and Google Play. Employees update on their own device management cadence. Customer cannot force app updates on employee-owned devices.
MDM / EMM management of FORM app If your organisation requires managed app distribution, your MDM team configures deployment. FORM does not integrate with MDM management planes.
§4

Application & Platform Security

Authentication, vulnerability management, and the CV computer-vision pipeline.

§4.1 · Application authentication
Control FORM Shared Customer
SSO integration (SAML 2.0 / OIDC) FORM maintains the SP-side implementation — assertion validation, PKCE, token issuance.
IdP configuration FORM provides configuration guides per IdP in docs/SSO_CLIENT_CONFIG.md. Customer IT configures the IdP application. Natively tested: Okta, Azure AD / Entra ID, Google Workspace, OneLogin.
Certificate rotation FORM rotates signing certificates 90 days before expiry and publishes updated metadata. Customer IT must update the SP certificate in IdP configurations that do not support automatic metadata refresh (e.g. ADFS). FORM notifies 60 days in advance.
Admin API key management FORM issues API keys; keys are hashed at rest (HMAC-SHA256). Keys age out after 365 days with soft enforcement (CSM notified). Customer is responsible for storing API keys securely. Leaked keys must be reported to security@form.coach immediately for revocation.
§4.2 · Vulnerability management
Control FORM Shared Customer
Secure SDLC Code review required before merge. SAST (CodeQL) in CI. No secrets in source (gitleaks pre-commit hook).
Annual penetration test Independent external pen test annually. Findings remediated per severity SLA (docs/PENETRATION_TESTING.md). Summary shared with enterprise customers under NDA.
Third-party integrations (customer-built) Customer is responsible for securing any tools that access FORM via the Admin API. OAuth tokens issued to customer-operated integrations are the customer's responsibility to rotate and revoke.
§4.3 · Computer vision (CV) pipeline
Control FORM Shared Customer
On-device inference CV models run on-device. Raw camera frames are never uploaded. Only derived pose keypoints are transmitted.
Model integrity Models are signed. The app validates the signature before loading.
Camera permission FORM requests camera permission only when CV coaching is initiated, using the OS-native permission prompt. Enterprise IT may restrict camera access via MDM policy — this will disable CV coaching for those devices (graceful degradation to rep-counting).

Identity & Access Management

User lifecycle, RBAC, and MFA/session policy ownership.

FORM enterprise roles
Role Permission scope Who assigns
tenant_owner Full org control — billing, SSO config, delete org FORM enterprise-architect only. Cannot be mapped via SCIM or SSO group — a hard schema constraint preventing IdP compromise from becoming a full org takeover.
tenant_admin All admin capabilities except delete org and billing transfer Customer IT via SCIM group mapping or manual assignment.
tenant_manager Read-only aggregate reports; manage assigned user groups Customer IT via SCIM group mapping.
member Personal coaching access only All provisioned users by default.
§5.1–5.3 · User lifecycle, RBAC & session policy
Control FORM Shared Customer
User provisioning (SCIM 2.0) FORM maintains the SCIM v2 endpoint. JIT provisioning creates records on first successful SSO assertion when enabled. Customer IT configures the SCIM connector in their IdP. FORM provides per-IdP guides. Customer controls whether JIT is enabled — regulated environments typically require SCIM-first provisioning.
User deprovisioning FORM revokes all active sessions synchronously when a SCIM deactivation signal arrives. Customer IT is responsible for removing the user from the IdP group or disabling their IdP account to trigger SCIM deactivation. Deprovisioning does not happen automatically on HR system termination unless HR is connected to the IdP.
Role-to-group mapping Role schema and enforcement in JWT claims + database RLS. FORM provides the mapping interface. Customer IT defines which IdP groups map to which FORM roles. On multiple conflicting groups, higher privilege wins (DEC-043 rationale).
Principle of least privilege Customer is responsible for granting only the minimum necessary FORM role to each employee group.
Access review (quarterly) FORM provides the audit log viewer and SCIM roster API. Customer's security or IT team is responsible for running periodic access reviews. FORM recommends quarterly.
MFA enforcement SSO is the only authentication path once configured. MFA enforcement is the IdP's responsibility. Customer IdP administrator enforces MFA on the IdP application. FORM strongly recommends MFA and will flag its absence at onboarding. FORM cannot enforce MFA independently of the IdP.
Session duration FORM enforces the maximum session duration configured for the tenant (session_timeout_hours, default 168h / 7 days). Sessions expire unconditionally at the configured maximum. Customer specifies timeout during onboarding. Regulated environments (healthcare, finance) commonly require 8 hours.
Employee ability to revoke company association Any employee can disassociate their FORM account at any time from Settings → Privacy → Leave organisation. FORM enforces this unconditionally.
§6

Data Security & Privacy

Encryption, multi-tenant isolation, and GDPR controller–processor boundary.

§6.1 · Encryption
Control FORM Shared Customer
Encryption in transit TLS 1.2+ on all endpoints. Certificate managed by Cloudflare. HSTS enforced.
Encryption at rest (Postgres + R2) AES-256 at rest on Supabase Postgres (Supabase SOC 2 evidence available) and Cloudflare R2.
Health data fields (GDPR Art. 9 special-category) Column-level AES-256-GCM on body_metrics, workout_sessions, ed_screening. Key rotation per docs/KEY_ROTATION.md.
Customer-managed encryption keys (BYOK) Not currently available. Roadmap: post-Series A. Enterprise customers requiring BYOK must note this as a contractual gap.
§6.2 · Multi-tenant data isolation
Control FORM Shared Customer
RLS policy enforcement Every application query runs under a Postgres role that has RLS enforced. form_admin (BYPASSRLS) is restricted to migration runner and compliance toolchain — never the API request path.
tenant_id sourced from JWT claims tenant_id is extracted from the verified JWT, never from client request parameters. A missing or invalid tenant_id returns 0 rows — fail-closed.
Cross-tenant penetration test Annual penetration test includes explicit cross-tenant isolation testing. A failed RLS bypass is a P0 finding with immediate remediation. Adversarial CI suite also tests this on every deploy.
Admin API key scope Admin API enforces tenant scoping at all endpoints. Customer is responsible for the security of Admin API keys. A compromised key can read all data within that tenant's scope.
§6.3 · GDPR controller–processor responsibilities
Obligation FORM (Processor) Customer (Controller)
DPA execution FORM provides a GDPR-compliant Data Processing Addendum. Executed before pilot data flows. Customer legal must review and sign the DPA before any employee data enters FORM.
Lawful basis for processing Customer must establish and document a lawful basis under Art. 6 — typically Art. 6(1)(b) (employment contract performance) or Art. 6(1)(a) (consent), depending on programme design.
Art. 13–14 employee notice FORM's consumer privacy notice (form.coach/privacy) covers data processed after first login. Customer must issue a separate Art. 14 notice to employees about SCIM-provisioned data (directory attributes synced before first login). FORM's privacy notice does not cover this.
Standard Contractual Clauses (SCCs) FORM includes SCCs Module 2 (controller-to-processor) in the DPA for US-region tenants processing EU employee data. Customer's legal team must confirm that SCCs are an appropriate transfer mechanism. Some sector-specific rules impose stricter requirements.
Sub-processor disclosure FORM maintains a public sub-processor list at form.coach/subprocessors. 30-day advance notice for material changes. Customer has the right to object to new sub-processors within the 30-day notice window.

Audit Logging & Evidence

FORM maintains an append-only, HMAC-chained audit log for all privileged actions (DEC-030). Any modification to a historical log record breaks the chain and is detected.

§7.1 · What FORM logs
Log category Retention Who can read Notifies tenant
auth.* — login, logout, SSO assertions, MFA events 3 years Tenant tenant_admin + FORM security-engineer No
admin.* — provisioning, role changes, config changes 7 years Tenant tenant_admin + FORM security-engineer Visible in audit log viewer
support.* — FORM employee views tenant data 7 years FORM security-engineer + compliance-officer ✓ Tenant admin email within 24h
data.* — sensitive field access, bulk exports, DSARs 7 years FORM compliance-officer Available on DSAR request
security.* — rate limit hits, quota warnings, break-glass access 3 years FORM security-engineer + tenant tenant_admin Alert at quota 80% / 95%
billing.* — contract events, seat changes, pilot state 7 years Tenant tenant_owner + FORM billing No
§7.2 · Audit log access responsibilities
Control FORM Shared Customer
Generating and storing audit logs All logs generated automatically. Cannot be disabled by tenant config.
HMAC chain integrity validation Validated on read and by scheduled nightly job. Integrity failure triggers a P0 incident.
Exporting audit logs FORM provides /v1/audit-logs export endpoint for tenant admins. SIEM integrations (Datadog, Splunk) and S3 sync available on Growth+ tier.
Reviewing logs for anomalies Customer's security team is responsible for reviewing their tenant's audit logs. FORM provides the data; FORM does not perform threat hunting on behalf of customers.
Retaining logs beyond FORM retention periods If customer compliance requirements exceed FORM's 7-year retention, customer must export and store logs in their own SIEM/archive before FORM retention elapses.

Operations & Incident Response

Uptime, SLA, and incident response ownership by severity classification.

P0 — Critical · data breach or full service outage
< 1h FORM response
Step FORM Customer
Incident detection Automated alerting (PagerDuty, 60-second probe intervals). May also report via security@form.coach.
Incident declaration and triage FORM security-engineer / devops-lead declares. Full internal triage by FORM.
Customer notification Tenant technical contact notified within 1 hour of declaration.

Employee / HR communication Customer HR / People team.

P1–P3 · Degradation and configuration incidents
Control FORM Shared Customer
P1 — Significant degradation or vulnerability FORM leads response. Tenant technical contact notified within 4 hours. Workaround communicated via CSM. FORM provides GDPR assessment. Shared with customer if personal data is at risk.
P2 / P3 — Customer-controlled configuration issues Provides diagnostic tooling — audit logs, SCIM sync status API — and support within standard SLA window. Joint diagnosis: FORM confirms the infrastructure is healthy; customer IT investigates IdP / SCIM configuration. Customer IT leads remediation on the IdP side (e.g. misconfigured SCIM, group mapping change).
Uptime SLA 99.9% monthly uptime (Growth and Enterprise). 99.5% (Starter). Credits calculated and issued automatically. Full terms in docs/ENTERPRISE_SLA.md. Customer may dispute credit calculations within 30 days of incident month close.
Disaster recovery Supabase PITR — 7-day restore window, RPO ≤ 1h. RTO target ≤ 4h for Supabase primary failure. Annual DR drill documented. FORM provides data egress API. Customer must initiate bulk export before offboarding deadline and retain data in own storage.

Compliance & Regulatory Obligations

FORM does not assess, verify, or warrant that using FORM satisfies any industry-specific compliance requirement.

§9.1 · FORM's compliance obligations
Area FORM's obligation Evidence available
SOC 2 Type II Target completion 12 months post-commercial launch. Continuous monitoring via Drata. SOC 2 report under NDA on completion. Pen-test summary + Drata dashboard access available in interim.
GDPR (as processor) DPA executed before data flows. Sub-processor list maintained. DSAR tooling provided. SCC in DPA for EU-US transfers. docs/GDPR_DPIA.md, docs/SUBPROCESSORS.md.
Penetration testing Annual external pen test. Critical findings remediated within 30 days. Summary shared under NDA. Full report available under NDA + MSA.
Cryptographic controls FIPS-aligned cipher suites. Key rotation policy and annual rotation schedule. docs/CRYPTOGRAPHY_POLICY.md, docs/KEY_ROTATION.md.
§9.2 · Customer's compliance obligations
Obligation Guidance
HIPAA (if applicable) FORM is not a Covered Entity and does not accept HIPAA BAAs. Enterprise customers in US healthcare must evaluate whether employee wellness data constitutes PHI. FORM's privacy floor minimises HIPAA exposure but does not eliminate it. Consult legal counsel.
ISO 27001 supplier assessment Customers conducting ISO 27001 assessments may request FORM's ISMS documentation summary, pen-test executive summary, and SOC 2 bridge letter under NDA.
FedRAMP / StateRAMP FORM does not hold and does not plan to pursue FedRAMP authorisation. Government sector customers requiring FedRAMP-authorised products cannot use FORM at this time.
Employee consent programmes Where the customer's jurisdiction requires employee consent before collecting wellness data, the customer is responsible for obtaining, recording, and honouring that consent. FORM provides a technical consent flow in the app; whether it is legally sufficient depends on jurisdiction and programme design.
Works council / union consultation (EU) In EU jurisdictions where employee health monitoring requires works council approval, the customer is responsible for obtaining that approval before deploying FORM.
Industry-specific data residency FORM offers EU (Frankfurt) and US (Virginia) regions. Country-level residency (e.g. Germany-only) is not currently available. Customers with this requirement must disclose it before signing.

Step ownership from contract signature to full rollout — and from termination to data deletion certificate.

§10.1 · Onboarding responsibilities
Step FORM Customer
DPA and MSA execution FORM prepares DPA and MSA. DPA must be signed before any pilot data enters FORM. Customer legal executes both documents.
Tenant provisioning FORM enterprise-architect provisions the tenant in production (slug, region, plan).
SSO / SCIM configuration Provides step-by-step guides per IdP. CSM available for live support on the setup call. Customer IT implements the IdP configuration and tests SSO in staging before pilot launch.

§10.2 · Offboarding responsibilities
Step FORM Customer
Data export window Data held for export for 30 days after contract end date. Customer must initiate bulk export before the 30-day window closes.
Data deletion Hard delete of all tenant data within 30 days of export window close. Written deletion certificate issued on request (satisfies GDPR Art. 17).
Art. 9 health data (DEC-036) Hard-deleted immediately — no grace period. This invariant cannot be overridden by MSA clause. Employees requiring personal data export must exercise portability (Settings → Privacy → Export) before the offboarding trigger date.
SSO / SCIM decommissioning Customer IT decommissions the FORM application from their IdP after offboarding to prevent residual access.
Audit log retention (post-offboarding) Audit log entries retained for the regulatory retention period (7 years for financial and admin events), in anonymised form.
§11

Liability Boundaries

When something falls in a gap, and controls FORM cannot provide at any contract tier.

§11.1 · Escalation path for control gaps
1
Customer security team raises the issue with their FORM CSM (Growth/Enterprise) or support@form.coach (Starter).
2
FORM's enterprise-architect and security-engineer jointly assess whether the control gap is on FORM's side, the customer's side, or genuinely shared.
3
Written gap assessment provided within 5 business days.
4
If gap is on FORM's side: internal issue filed, added to remediation backlog with a target date. If on customer's side: FORM provides a configuration guide and CSM support. If genuinely shared: both parties agree on a remediation plan.
§11.2 · Controls FORM cannot provide
Out-of-scope control Reason
MFA enforcement on the SSO path Controlled by the customer's IdP. FORM cannot force-enable MFA on a third-party identity provider.
Physical device security for employee devices FORM is a mobile app. Physical security is outside FORM's perimeter.
Preventing insider threat from within the customer's IdP admin team A compromised IdP admin can provision rogue users. FORM mitigates this by requiring FORM enterprise-architect approval for tenant_owner changes, but cannot fully prevent an IdP-level compromise.
Monitoring employee use of the FORM app FORM's privacy floor prohibits employer access to individual user data. This is a product constraint, not a technical gap.
HIPAA BAA Not available at this time.
Country-level data residency below EU / US region Not available at this time. Roadmap: post-Series A.
§12

Non-Negotiable Floors

These customer requests are declined regardless of contract value or legal pressure. These protections exist for employees — FORM's end users — and cannot be waived by any party.

Declined regardless of contract value or legal pressure
Request Response
Grant HR access to individual employee health, body composition, or coaching data Declined. Privacy floor enforced at application + database layer. Not configurable by any admin role or MSA clause.
Share workout performance or health data for insurance premium calculation Declined. Explicitly listed in docs/ENTERPRISE.md §When we say no.
Enable real-time monitoring of individual employee fitness activity Declined. Admin dashboard shows aggregates only, with a minimum cohort size of 10. No individual time-series is exposed.
Provide a backdoor or law enforcement access without valid legal process Declined. FORM follows a legal-review-first government request policy. Narrowest possible response. User notification where legally permitted. Transparency report published annually.
Tie wellness participation to performance reviews, insurance, or employment consequences Declined. Employee participation data is never surfaced to managers in any form. A contract that requires this is not a contract FORM signs.
Custom feature that weakens the HMAC audit chain (DEC-030) Declined. Audit chain integrity is a SOC 2 Type II control. Weakening it invalidates the control for all tenants and the observation period.
§13

Glossary

Processor
FORM's legal role under GDPR. Processes personal data on behalf of the controller.
Controller
The enterprise customer. Determines the purpose and means of processing employee data.
DPA
Data Processing Addendum. Contract between controller and processor per GDPR Art. 28.
SCIM
System for Cross-domain Identity Management. Protocol for automated user provisioning.
RLS
Row-Level Security. Postgres feature that enforces tenant data isolation at the database layer.
JIT
Just-In-Time provisioning. User record created in FORM on first successful SSO login.
HMAC chain
Hash-based Message Authentication Code chain. Provides tamper-evident audit log integrity (DEC-030).
Break-glass
Emergency access procedure for FORM personnel to access a tenant's data with 2-person approval and time-bounded scope.
CSM
Customer Success Manager. Named FORM contact for Growth and Enterprise plan customers.
DSAR
Data Subject Access Request. Formal request by an individual to receive or delete their personal data.
Privacy floor
Non-negotiable data access boundaries protecting employees from employer surveillance. Enforced at application and database layer.
SOC 2 Type II
Audit of security controls over a 12-month observation period by an independent CPA firm.
SCCs
Standard Contractual Clauses. EU-approved contractual mechanism for transferring personal data from the EU to third countries. Included in FORM's DPA.
